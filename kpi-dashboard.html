Campus Breakdown — Follow-Up Metrics by Campus Type

Beta Follow-Up campuses are using a specific follow-up workflow in Rock that other campuses are not yet on. All campuses can follow up with people inside Rock using existing workflows, so the Beta workflow alone would not necessarily be expected to improve completion rate or response time broadly.

New Family Registration — The connection type most likely to be impacted by the Beta workflow, since new families feed directly into the multi-step follow-up flow. This is the metric to watch for a meaningful difference between Beta and Non-Beta campuses.
LifeGroup Interest — Higher completion rates (~60% YTD) are driven by SMS notifications sent directly to the LifeGroup leader, plus an escalation rule that routes to the community pastor if unhandled. Multiple accountable people receiving direct notifications is what's moving this metric, not the Beta workflow.
Baptism — Received a workflow refresh last year. Any improvement in this connection type should be read in that context rather than attributed to the Beta Follow-Up workflow.

Open Questions

Is the build time we care about the AppVeyor build of Rock itself, or our total pipeline time including our customizations on top of Rock? These could tell very different stories — the former is partially outside our control, the latter is fully owned by our team.

Where do our automated tests actually run — in GitLab CI, or are they tied to the AppVeyor build? This determines where coverage % and pass rate data lives and how we pull it.

Do we have access to AppVeyor build history and API to pull build duration data programmatically, or will this need to be logged manually per upgrade?

Version overlap policy: when a new major version releases while the previous version is still receiving point releases (e.g. v17 point releases continuing after v18.1 dropped in Dec 2025), what is the official rule for when the upgrade clock starts? Need a defined policy so kickoff lag is measured consistently going forward.
